1What we are doing

Midnight Passport gives a user a named, self-sovereign, privacy-preserving identity on the Midnight network — without ever seeing a seed phrase, a raw address, or a gas-token purchase screen. The project's north star is the secure-onboarding-design authored by Charles Hoskinson, which sets out six design principles for how identity, custody, and signing should behave in the ecosystem.

The six principles, quoted briefly from the source:

  1. Keys never leave TEEs; authorisation via ZK witnesses.
  2. Single seed, three wallets (Shielded, Night, Dust).
  3. Chain abstraction via intent model.
  4. One key per device.
  5. Privacy-by-design identity.
  6. Seedless UX.

Delivering all six principles in their final form in one go is not realistic; several are open research areas in their own right. The design also has implications well beyond the ARC boundary. We have therefore been engaging with the wider ecosystem from the start.

Who we have been talking with

  • Midnight Foundation — aligning on the platform surface (curves, proof pipeline, APIs, CAIP namespace), the Midnight Improvement Proposal (MIP) process, and ensuring the architecture we build is one the Foundation recognises as the Midnight-native direction.
  • Lace — the team building Lace ID. Their existing key-derivation and address implementation directly informs the first two standards we plan to publish: the key-derivation MIP and the Bech32m address-format MIP. Their ID product is also a natural consumer of the credential layer we will add later.
  • Cryptographic reviewer (Jesus Diaz Vico) and formal methods (James Chapman's team) — per-phase review and long-running rigour, respectively.

How we plan to get there

We cannot build everything at once. The plan is:

  1. Pick a subset of the principles to deliver fully in a first MVP.
  2. Accept honest, explicit compromises on the rest — not drop them, but place them on a named later step.
  3. Start building immediately, with a weekly demo as the unit of progress. Every Friday, one artefact that has moved. Over a few months, enough artefacts accumulate that they compose into the MVP.
  4. Use each weekly update both for credibility and for concrete evidence — we show a working thing, we say what did not work, we commit to next week's artefact.

Parts of the plan are already de-risked. Internal experiments have validated that Schnorr signatures verify inside a Compact circuit on a local Midnight devnet, and that Rust and TypeScript signing sides produce identical outputs across the language boundary — so the signing primitive the current MVP path depends on is not speculative.

Why this matters beyond Passport

The ultimate goal is not an isolated wallet. It is a set of protocols and standards — published as MIPs — that the rest of the Midnight ecosystem can build on. With these in place, any wallet, any dApp, any credential issuer can plug into the same names, addresses, signatures, connection protocol, and credential system, without inventing its own. That interoperability is what turns Midnight Passport from an IOG product into a piece of ecosystem infrastructure — and what turns a user's privacy-preserving, self-custodial, multi-device experience from a bespoke app feature into a property of the network itself. User-friendly by default.

The north star, in one sentence

Define the identity, custody, and connection primitives of Midnight so that a user lands on a named account via a passkey, moves across the network and its dApps in a way that is privacy-preserving, self-custodial, interoperable — and effortless.

2Delivery horizon

Getting from today to the end-state takes roughly a year and change. The scope of that end-state is set by the design; the sequence is set by one rule: deliver user-facing value in the first MVP, then strengthen and extend without changing the user experience.

Each step below is an independently demoable product. The user experience is stable across steps — what improves underneath is decentralisation breadth, multi-device support, recovery, and cross-chain. We target roughly a 90-day beat between major steps after the MVP.

Why UX first

A plan that holds the user experience constant across every step is the plan that can keep stakeholders aligned for a year. At each step, the user sees the same product getting better at what it already does — multi-device support, recovery, credentials, cross-chain. No step forces a re-onboarding. No step feels like a reset.

Plan updated 2026/04/21

The current MVP path makes the user's passkey the signing root. All cryptographic operations run client-side in the browser; the account provider is a lighter service — JWT issuance, name resolution, rate limiting, optional sponsorship — with no custody authority. The NightExternal intent signature verified at the Midnight node level is sufficient authorisation for Compact contract calls, including name registration: MVP 1 performs no in-circuit signature verification, and the 1-of-N in-circuit Schnorr primitive preserved in the experiments returns under MVP-09 as the multi-device foundation. The trade-off is a limited browser / OS matrix at MVP and account recovery deferred behind genuine multi-device support. The original plan (Plan B — FROST threshold committee, universal browser support, federation-mediated recovery) is preserved as the foundation for Milestone 2 cross-chain and as the fallback. See how the plan evolved.

2.1MVP 1 — end of June 2026

MVP 1 Apr → end of June 2026 · demo target: end of June
The smallest product that delivers on the spirit of the design — decentralised from day one

Principles we deliver

  • Principle 6 · Seedless UX — DELIVERED. Passkey (WebAuthn) authentication from day one. The user never sees a seed, a mnemonic, or a raw address. Stronger than before: no seed is ever generated or held by a third party — the passkey is the signing root.
  • Principle 1 · Authorisation via ZK witnesses — SHIFTED. The ZK-witness authorisation model is fully in place. The "TEE" is the browser's passkey-backed credential store (WebAuthn platform authenticator). The Schnorr signature is consumed as a private witness, exactly as in the reference experiments.
  • Principle 2 · Single seed, three wallets — DELIVERED. A passkey-derived root feeds CIP-1852 derivation for Shielded, Night, and Dust. The seed never leaves the browser; the user holds it alone.

Principles we honestly defer

  • Principle 3 · Chain abstraction — deferred to Milestone 2+. Same deferral as in the original plan. Cross-chain is the natural home for the preserved threshold-signing substrate.
  • Principle 4 · One key per device — redefined at MVP as "one key per browser instance". MVP 1 is strictly one physical device, one account. Genuine Passport-engineered multi-device (two or more physical devices, device-add ceremonies, per-device keys) is a near-term priority after MVP 1 because it gates recovery.
  • Principle 5 · Privacy-by-design identity (credentials) — deferred to Milestone 3+. Independent of the custody model, so it can land later without architectural rework.

Compromises we accept (and name)

  • Limited browser / OS matrix at MVP. Passkey behaviour, key-material persistence, and WebCrypto availability vary across browsers and operating systems. MVP 1 commits to Browser A × OS-1 and Browser A × OS-2; the exact pairs are a product-owner-deferred decision.
  • No account recovery at MVP. If the user loses the passkey and has no platform sync, the account is gone. This is stated to the user at onboarding in plain language. Recovery cannot be built until genuine multi-device exists — the order is multi-device first, recovery second.
  • One physical device, one account. Platform passkey sync (Apple iCloud Keychain, Google Password Manager) may replicate a passkey across a single user's own devices where they have enabled it; this is convenient but is not Passport multi-device and is not relied on in the product design.
  • Private / incognito mode unsupported. The MVP surface refuses onboarding in private mode rather than creating an account that silently disappears at the end of the session.

What ships at the June demo

Passkey onboarding → browser-derived wallet keys (Shielded, Night, Dust) → name registration (alice.midnight) → first sponsored transaction on Midnight devnet. Plus the account-loss sad paths shown honestly. Plus "what's next" pointing to multi-device.

2.2Near-term post-MVP — multi-device and recovery

Post-MVP · near-term Brought forward · ahead of the original Milestone 2 slot
Genuine multi-device accounts and the path to recovery

Genuine Passport-engineered multi-device — two or more physical devices under a single account, with a device-add ceremony and per-device keys — is a near-term priority after MVP 1. It is brought forward ahead of the original Milestone 2 slot because it gates recovery. The MVP-09 label "research programme" still applies during MVP 1; exact delivery timing is refined in the next planning pass.

Likely shape: a per-device key registered under the user's on-chain account, with a device-add ceremony between the existing and new browser. This converges toward the multi-key account contract (MIP-4 in the preserved plan), which can be adopted wholesale once it exists.

Recovery design work feeds the multi-device programme. The order is firm: multi-device first, recovery second. Design space (sketched, not committed) covers DeRec-style social recovery and encrypted-blob backup; neither is in MVP 1.

2.3Milestone 2 — cross-chain and the connection protocol

Milestone 2 ~90-day cycle · begins ~July 2026
Cross-chain substrate and the dApp-wallet connection protocol

Milestone 2 delivers the dApp-wallet connection protocol as a formal MIP with named external co-authors — the Midnight equivalent of Cardano's CIP-30 — and introduces the cross-chain substrate. The preserved threshold-signing artefacts (Plan B work) become the natural foundation for cross-chain custody and inter-party federation — they were not wasted work.

Milestone 2 also consolidates the multi-key account pattern on-chain (the Stream A work from the original plan), moving device-to-account mapping from the account provider onto the chain.

2.4Milestone 3 — credentials, KYC, decentralisation endgame

Milestone 3 successive 90-day effort cycles
The additive layer on top of a decentralised foundation

With multi-device and cross-chain in place, Milestone 3 adds the features that build on top:

  • Privacy-preserving credentials MIP — Principle 5 delivered. Selective disclosure of identity attributes via zero-knowledge. Built on Midnight's attestation-tree primitive. Multi-issuer from day one.
  • KYC and compliance — verifiable credentials issued by trusted third parties, reusable across dApps without re-verification.
  • Cross-chain intent engine — Principle 3 delivered in full.
  • Social account linking — a vision-author priority, preserved and carried forward.
  • DeRec-based decentralised social recovery — the boss-priority recovery feature, built on the multi-device foundation landed earlier.
  • On-device cryptography — Principles 1 and 2 delivered in their strongest form; the browser-held root migrates into device TEEs (Secure Enclave, StrongBox) for platforms that support it.
Principle-to-date summary

Principles 6 (seedless), 2 (single seed, three wallets), and 1 (ZK-witness authorisation) land at MVP 1. Principle 4 (one key per physical device) lands near-term post-MVP via genuine multi-device. Principle 3 (chain abstraction) at Milestone 2. Principle 5 (privacy-preserving credentials) at Milestone 3. Every principle has a home in this plan.

3The phases to the MVP

Six live phases carry the MVP requirements. Phase 3 (threshold-signing cornerstone) is parked and preserved as Milestone 2 input; Phase 4 (in-circuit verification) is removed — the NightExternal intent signature verified at the node level is sufficient authorisation for MVP 1. A new Phase 3' covers the client-side signing work introduced by the current path. The overall schedule is now expected to be approximately three weeks lighter than the original plan.

Phase 1
Foundation — scope lock, delegation, weekly-demo contract, prototype-repo contract, evidence log. Covers META-01/02/03/05/07 and EVD-01. The ground rules everything else rests on.
Phase 2
Cross-cutting prerequisites — STD-03 domain-separation registry, the adoption narrative (META-04), and the formal-methods engagement plan (META-06). Priority-1 formal-methods target under the current path is the name-registry commit-reveal scheme with ENSIP-15 normalisation enforced in-circuit.
Phase 3
Threshold-signing cornerstone — Parked 2026/04/21, deferred to Milestone 2 cross-chain. Preserved in full as Milestone 2 input and as the fallback should the current MVP path hit a blocking risk. See the original Plan B for the full treatment.
Phase 3'
Client-side signing spec and browser support matrix — New. Covers MVP-08 (client-side signing of Midnight intents with the NightExternal key, passkey → seed → wallet-role keys, Path A via WebAuthn PRF and Path B assertion fallback) and the first slice of MVP-09 (research programme on passkey material stability, multi-device design space, and the 1-of-N in-circuit Schnorr verification pattern absorbed from the removed MVP-02). The MVP-08 browser × OS matrix is a product-owner-deferred decision.
Phase 4
In-circuit verification — Removed 2026/04/21. The NightExternal intent signature verified at the Midnight node level is sufficient authorisation for Compact contract calls, including name registration; MVP 1 performs no in-circuit signature verification. The 1-of-N in-circuit Schnorr primitive preserved in the experiments returns under MVP-09 as the multi-device foundation.
Phase 5
Account provider (simplified) and name registry — MVP-03 and MVP-04. The account provider is a thin sideband — WebAuthn, JWT issuance with cnf claim, rate limiting, optional sponsorship paymaster. No co-signing, no recovery hand-off, no key custody. Name registry contract unchanged from the original plan; ownership is bound by the NightExternal intent signature that the node already verifies.
Phase 6
Prerequisite MIPs — STD-01 (key-derivation paths, CIP-1852-aligned, coin type 2400) and STD-02 (Bech32m mn_ address format). Each opens as a PR against the Midnight Improvement Proposals repository with a named external co-author. STD-01 can start in parallel with Phase 3'.
Phase 7
MVP integration and UX walkthrough — MVP-05 and MVP-06. End-to-end onboarding spec (passkey registration → browser-derived keys → name registration → first sponsored transaction) plus the UX walkthrough covering happy path and account-loss sad paths (R1 platform passkey rotation, R2 browser data clear, R4 "account loss is possible — state this to the user") and private-mode refusal. MVP-07 removed: there is no signing service to operate, so the single-node operational model is dropped.

Dependency graph

Phase 2 depends on Phase 1. Phase 3' depends on Phases 1 and 2. Phase 5 depends on Phases 2 and 3' (the provider must reject any request whose cnf does not bind the browser-signed body). Phase 6 depends on Phase 2 for STD-03 prefixes; STD-01 can run in parallel with Phase 3'. Phase 7 consumes Phases 3' and 5. (Phase 4 is removed; the previous Phase 7 → Phase 4 dependency is dropped.)

Weekly-demo cadence

Every Friday, one artefact that moved. The weekly commit is the contract; the June MVP is the aspiration. If a week's intended deliverable slips, the rule is to show the next thing that works rather than a broken thing. Two consecutive slippages on the same critical-path item trigger a phase-boundary escalation.

Slippage protocol (applies to every week)

If the week's planned demo is not on track by Wednesday, the team must pick an adjacent deliverable (a smaller component, a predecessor step, or a parallel artefact from another lane) and show that Friday instead. Showing a working adjacent thing preserves the weekly rhythm; apologising for a missed week destroys it. Two consecutive slippages on the same critical-path item trigger an escalation at the next phase-boundary review.
