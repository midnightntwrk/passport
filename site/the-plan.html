The MVP architecture
03 / 07
Five building blocks · browser-first

What ships
by end of June.

All of the cryptographic work lives in the browser. The account provider is a thin sideband — names, JWTs, rate limits, optional sponsorship — with no custody authority and no key material.

Browser holds the root. Compact circuit verifies. Provider is a sideband. The MVP architecture is the Day-1 decentralised architecture — there is no scaffolding here that needs to retire.

From scope-lock
to MVP.

Six live phases carry MVP 1. Phase 3 (threshold signing) is parked and preserved as input to Milestone 2 cross-chain. Phase 4 (in-circuit verification) is removed: the NightExternal intent signature verified at the node level is sufficient authorisation for Compact contract calls, including name registration. Account provider is a thin sideband — WebAuthn, JWT issuance with cnf claim, rate limiting, optional sponsorship. No co-signing, no recovery hand-off, no key custody. Name registry contract unchanged — commit-reveal, multi-address resolution, ENSIP-15 in-circuit, per-device-key rate limits.

Current Phase 6 · Prerequisite MIPs Key derivation and address format STD-01 · STD-02

CIP-1852-aligned key-derivation paths for Midnight (coin type 2400). Bech32m mn_ address format across Shielded, Night, and Dust types. Each MIP opens as a PR against the Midnight Improvement Proposals repository with a named external co-author. STD-01 can run in parallel with Phase 3'.

Current Phase 7 · MVP integration MVP integration spec and UX walkthrough MVP-05 · MVP-06 · MVP-07 removed

End-to-end onboarding spec: passkey registration → browser-derived keys → name registration → first sponsored transaction. UX walkthrough covers the happy path and the account-loss sad paths (R1 passkey rotation, R2 browser data clear, R4 "account loss is possible — state this to the user"), plus private-mode refusal. Single-node operational model (MVP-07) is removed; there is no signing service to operate.

Phase 3 is parked, preserved as Milestone 2 input. Phase 4 is removed, folded into MVP-09. Phase 3' is new, covering the client-side signing work. Phases 5 and 7 shorten under the current path because there is no committee to operate. The June 2026 MVP target stands.

The current path is now expected to be approximately three weeks lighter than the original plan — another week lighter than the previous revision — because the FROST adoption, the in-circuit single-signer Schnorr verification, the single-node operational model, and the committee-wiring work have all been dropped from the MVP critical path.

Standards track
05 / 07
MIPs pipeline · parallel to build

Prerequisite MIPs
ship during the MVP.

Midnight has no formal improvement-proposal process yet; the Passport MIPs are both the vehicle for interoperability and the forcing function that creates that process. Every MIP ships with a named external co-author — unilateral drafts become shelfware.

Key-derivation and address-format MIPs ship during the MVP window, OWS-aligned where applicable. The connection protocol, credential standard, multi-key account, recovery paths, and DecentralisedAuth extend the pipeline across Milestones 2 and 3. MVP-08 (client-signing + browser matrix) is new under the current path.

Evidence
06 / 07
What's validated

Already running
on devnet.

The cryptographic plan is not speculative. Three core assumptions are already validated end-to-end on a local Midnight devnet. Under the current path, MVP 1 performs no in-circuit signature verification — the NightExternal intent signature is verified at the node level — but the in-circuit Schnorr primitive is preserved in the experiments and returns under MVP-09 as the 1-of-N multi-device foundation.

Authority-boundary invariant. The account provider cannot produce a Schnorr signature under any user's key. It holds no key material and no key share. Every user signature is produced in the browser, bound to the user's passkey, and verified by the Compact circuit as a private witness. This invariant is testable end-to-end and is the security cornerstone of the current path.

The MVP crypto plan rests on code that already runs. The remaining work is the client-side signing spec, the browser support matrix, the simplified provider, and the standards track — not novel cryptography.
