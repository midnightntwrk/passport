### Discussion with Claude Code about IP and licensing for the Midnight TUI

#### Prompt

> What is the situation for licensing code generated by LLMs such as yourself? Your wrote essentially all of the code under my guidance, but your training set contains material with many different license terms. How do we create a LICENSE file that respects all of the terms of the training data? How do we ensure that the resulting code is not considered plagarism? What sort of disclaimer shall we add to this folder?

#### Response

> This is genuinely unsettled legal and ethical territory. Let me walk through each dimension carefully.
> 
> ### Copyright status of the output
> 
> The core legal question is: **who (if anyone) holds copyright in AI-generated code?**
> 
> The US Copyright Office's current position is that purely AI-generated works are not copyrightable, but works with *sufficient human creative authorship* are. The boundary is not well-defined. In your case, you provided architectural direction, requirement specifications, review, correction, and rejection of approaches ŌĆö that is substantive creative input. Most copyright scholars would say you have at least a colorable authorship claim. The EU is similarly unsettled but trending toward requiring human authorship.
> 
> **Practical implication:** Assert copyright as your organisation's work. A court might disagree someday, but no one is likely to challenge it, and asserting it costs nothing.
> 
> ### Training data contamination
> 
> This is the most common concern, and largely theoretical for code of this type:
> 
> 1. **LLMs generate, they do not retrieve.** The model learns statistical patterns. Output is not fetched from a database of training examples ŌĆö it is synthesised token by token. No court has ruled that LLM output is a derivative work of training data.
> 
> 2. **Functional code elements are not copyrightable.** API call patterns, algorithm structures, and common idioms are not protectable. The vast majority of what makes code useful is functional, not expressive.
> 
> 3. **This code is highly specific.** The Midnight SDK is recent enough that it was minimally represented in training data. The TUI's architecture ŌĆö the hook/screen separation, the state machine patterns, the DUST monitoring heuristic ŌĆö reflects your design decisions, not reproduced training examples.
> 
> 4. **The ongoing litigation** (Andersen v. Stability AI, the GitHub Copilot class action) has not produced rulings establishing that LLM outputs infringe training data copyrights. Early motions have generally not gone well for plaintiffs on the direct copying theories.
> 
> The GPL-contamination fear specifically ŌĆö that training on GPL code makes all output a GPL derivative ŌĆö has no legal support and is widely regarded as implausible even by critics of AI training practices.
> 
> ### Is it plagiarism?
> 
> Plagiarism is an academic/ethical concept, not a legal one, and it requires presenting *someone else's specific work* as your own. There is no specific work to attribute here. A closer analogy: a human engineer who learned React by reading open-source code, then wrote a new component from memory, is not plagiarising ŌĆö even though their mental model was shaped by reading others' work. The AI is a sophisticated tool in the same epistemic position.
> 
> The honest acknowledgment is simply: **the code was written with AI assistance**. That is not plagiarism; it is transparency.
> 
> ### Ethics of public release
> 
> The ethical concerns worth taking seriously:

> - **Open-source community norms.** Some developers who licensed their work as GPL did so expecting derivatives to remain GPL. If LLM outputs are eventually found to be GPL derivatives (unlikely but non-zero), releasing under Apache 2 would violate that intent even if not the letter of current law. ### ­¤æ▒­¤ż¢ Midnight TUI ŌĆö `experiments/mn-tui`

`mn-tui` is a terminal-based wallet and contract tool for the Midnight network, built with [Ink](https://github.com/vadimdemedes/ink) (React for the terminal) and the Midnight wallet SDK. It provides a keyboard-driven interface for day-to-day operations on preprod/preview/mainnet/local networks without requiring a browser wallet: syncing balances, sending NIGHT and custom tokens, minting and deploying contracts, registering NIGHT for DUST generation, and managing an encrypted multi-wallet keystore. It can be run directly from this repository with `nix run .#mn-tui` (no local Node.js install required) or via `npm start`; on first launch it opens on the Network Configuration screen.

Capabilities by screen:

- **Dashboard** ŌĆö live chain status (block height, epoch, slot, peers, sync state) alongside wallet addresses, unshielded/shielded token balances, and a DUST Generation monitor showing accrual rate, max cap, estimated fill time, and a cross-wallet-registration warning when DUST is unexpectedly not accruing.
- **Send** ŌĆö batch one or more transfers of NIGHT (unshielded or shielded) or any non-zero contract token to arbitrary recipient addresses; shows live ZK proof and submission status.
- **Mint / Deploy** ŌĆö mint shielded fungible tokens to the active wallet via an existing or freshly-deployed fungible-token contract; also supports deploying arbitrary compiled Compact contracts with optional custom witnesses.
- **Keys** ŌĆö multi-wallet keystore: add wallets from a 24-word BIP-39 mnemonic (passphrase-encrypted with OpenPGP), unlock, activate, delete, and clear the sync state cache; addresses are stored and derived per-network (Midnight Bech32 addresses encode the network ID).
- **Designate** ŌĆö register or deregister all unregistered NIGHT UTXOs for DUST generation; the DUST receiver address can be directed to any wallet, with a warning when cross-registration is detected by a rolling 60-second balance-trend heuristic.
- **Network** ŌĆö select mainnet / preprod / preview / undeployed and configure per-network node, indexer, and proof-server URLs; settings are persisted across sessions.
- **Contract** ŌĆö query the `total_supply` and `mint_nonce` of any deployed fungible-token contract via the indexer; supports refresh.
- **Logs** ŌĆö tail the last 40 log entries (INFO/WARN/ERROR) with a new-entry badge in the nav bar; log path is configurable.

See [`experiments/mn-tui/lessons-learned.md`](../experiments/mn-tui/lessons-learned.md) for a record of non-trivial SDK obstacles and workarounds encountered during development.

### ­¤ż¢ New assessment: sharding in UTXO-based blockchains

Added [`assessments/utxo-sharding.md`](../assessments/utxo-sharding.md), surveying the major approaches to sharding UTXO chains and evaluating their applicability to Midnight.

Key findings:

- **The core UTXO sharding problem** is cross-shard atomicity: a transaction consuming an input from shard A and creating an output in shard B must be atomic without a global coordinator. This problem does not arise in the same form in account-model sharding (e.g. NEAR Nightshade).
- **Two-phase commit** (OmniLedger, Kronos) is the standard academic approach: input shard locks the UTXO, issues a proof-of-lock, output shard creates the new UTXO, both acknowledge. Provably atomic but adds two inter-shard round-trips per cross-shard transaction; vulnerable to replay attacks without embedded shard IDs.
- **Alephium BlockFlow** is the only significant production UTXO sharding system: G┬▓ chains (G=4 on mainnet), DAG causal ordering replaces two-phase locking, single-step cross-group UX, ~20,000 TPS. Address-group assignment is static; load imbalance has no clean mitigation.
- **Parallel execution** (intra-shard, not true sharding) is natively applicable to Midnight: independent ZK proofs can be verified on separate CPU cores without any cross-shard coordination.
- **State channels / Hydra** are the most applicable L2 sharding-equivalent for Midnight: isomorphic eUTXO channels, linear scalability per Head, maps to the Midnight City pattern.
- **ZK proof verification is atomic and indivisible**: sharding multiplies transactions processed system-wide but does not reduce the per-transaction ZK cost on any shard's validators. This makes L1 sharding a poor fit for Midnight compared to proof folding (ProtoGalaxy), which reduces the per-transaction cost globally.
- **Conclusion for Midnight:** ProtoGalaxy folding dominates L1 sharding as a throughput lever; parallel execution and state channels are the applicable sharding-adjacent patterns.

### ­¤ż¢ New assessment: TEE as a privacy fast path

Added [`assessments/tee-cheat-codes.md`](../assessments/tee-cheat-codes.md), examining TEE usage in NEAR and Midnight and evaluating where TEE integration could serve as a near-term privacy accelerator for Midnight, motivated by Charles's "cheat codes" framing in Background 6.

Key findings:

- **NEAR's TEE pattern:** TEE attestation is used in Chain Signatures (`v1.signer`) to prove MPC nodes are running unmodified signing code; TEE handles *execution integrity* while FROST/MPC handles *key custody*. Neither alone is sufficient.
- **Midnight has no TEE today:** privacy is purely ZK-based (Plonk/KZG); private witnesses never leave the client and no attestation infrastructure exists.
- **Privacy model contrast:** ZK offers unconditional mathematical guarantees; TEE offers conditional guarantees (trust hardware manufacturer + firmware supply chain). TEE is substantially faster to deploy; ZK is substantially stronger.
- **Highest-priority integration ŌĆö remote proof server:** A TEE-attested remote prover would enable mobile and browser dApp use cases; TEE protects the proving side while ZK still protects on-chain privacy. Engineering cost over a non-attested remote prover is modest.
- **Medium-term ŌĆö MPC / Chain Signatures analogue:** TEE + FROST threshold signing to provide a Midnight-native cross-chain signing service. NearFall spec ┬¦2.1.5 confirms the BN254/BLS12-381 incompatibility is confined to Sirius (the proposed IVC aggregation layer), not the production signing path ŌĆö `near/mpc` has no BN254 dependency. This unblocks Option 2 code reuse at the signing layer.
- **New blocker identified:** A separate signing-path incompatibility exists (NearFall spec ┬¦1.2.3): NEAR provides threshold ECDSA; Midnight requires BIP-340 Schnorr. These are distinct schemes ŌĆö ECDSA MPC protocol cannot be reused for Schnorr signing.
- **The "cheat codes" synthesis:** deploy TEE for capabilities where ZK circuits don't exist yet; build ZK paths in parallel; upgrade from TEE to ZK as circuits mature. Consistent with how Zcash and others have evolved their proof systems over time.
- Four outstanding questions identified for a dedicated next-increment TEE assessment: TEE platform selection, threshold Schnorr protocol for Midnight (primary remaining blocker), on-chain attestation verification cost, regulatory acceptability, and DUST/fee model interaction.
- ­¤¦¬ **Hypothesis 3** (new): BN254 incompatibility is confined to Sirius aggregation; `cait-sith` + FROST signing code is adoptable for Midnight's MPC network without curve work.

### ­¤ż¢ New assessment: batch proof verification vs. ProtoGalaxy folding

Added [`assessments/batching-vs-folding.md`](../assessments/batching-vs-folding.md), contrasting the two main ZK proof aggregation techniques applicable to Midnight.

Key findings:

- **Batch verification** (verifier-side): collapses k pairing evaluations to O(1), leaving MSMs sub-linear; net ╬│_txs improvement ~2ŌĆō5├Ś; ╬▓_txs unchanged (all k proofs still occupy block space); no new infrastructure or circuit changes; requires only a safe randomness source for the batch host function.
- **ProtoGalaxy folding** (prover-side): an aggregator collapses k same-circuit proofs into one before the block; both ╬│_txs and ╬▓_txs improve by ~k├Ś; requires aggregator infrastructure, folding-relation circuit design, and a security audit (­¤øæ BLOCKER).
- **Composition**: the two are complementary ŌĆö ProtoGalaxy reduces k circuit-A proofs to one folded proof; batch verification then combines the small number of folded proofs from different circuit types with one pairing check.
- **Near-term recommendation**: batch verification is deployable as a host function upgrade and provides partial ╬│_txs relief while ProtoGalaxy remains audit-blocked.

### ­¤ż¢ New assessment: throughput improvement hypotheses

Added [`assessments/throughput-hypotheses.md`](../assessments/throughput-hypotheses.md), quantifying the isolated and combined TPS contribution of six throughput techniques for Midnight.

Key findings:

- **Block size** and **slot time** are the only direct TPS multipliers: 200 KB ŌåÆ 5 MB gives ~15├Ś in isolation (compute-limited at ~49 TPS); 6 s ŌåÆ 2ŌĆō3 s gives ~2ŌĆō3├Ś in isolation.

### ­¤æ▒­¤ż¢ Named-Account Registry: Stealth Meta-Address PoC on Midnight

Designed, implemented, debugged, and confirmed a full end-to-end proof of concept for NEAR-style named accounts on Midnight. A registered name (e.g. `alice.midnight`) maps to a stealth meta-address `(K_scan, K_spend)` stored in a Compact contract; senders use those public keys to derive one-time shielded addresses without revealing a static destination to chain observers. Full design notes in [experiments/compact-named-accounts/design-notes.md](../experiments/compact-named-accounts/design-notes.md).

#### Contract design

The Compact contract (`single-named.compact`) exposes three public ledger groups and four circuits:

- **Registration:** `register(name, K_scan, K_spend)` ŌĆö stores the meta-address on-chain.
- **Unshielded escrow:** `send(r, P, amount)` and `claim(k_scan_scalar, k_spend_scalar, h)` ŌĆö ZK-proven deposit and withdrawal using the one-time address `P = K_spend + h┬ĘG`.
- **Shielded bulletin board:** `publish_ephemeral(r)` ŌĆö stores `R = r┬ĘG` so Alice can derive the stealth seed off-chain; the actual token transfer is performed by the wallet SDK outside the circuit.

#### API impediments discovered

Several Compact/SDK API gaps required workarounds:

1. **`EmbeddedFr` decode failure.** `Bytes<32> as Field` gives a BLS12-381 Fr value (~255 bits); `ecMulGenerator` requires a Jubjub scalar (EmbeddedFr, ~252 bits). Values exceeding `JUBJUB_R` (~94% of random 32-byte samples) cause a runtime decode failure after passing type-checking. Workaround: compute `h = persistentHash(S) % JUBJUB_R` in TypeScript and pass as a private witness; circuit verifies via `ecMulGenerator(h)`.
2. **No `hashToScalar` primitive.** There is no built-in way to reduce a `Bytes<32>` hash to the Jubjub scalar field inside a circuit.
3. **No modular arithmetic on `Bytes`.** `Bytes<32>` cannot be reduced modulo an arbitrary constant inside Compact.
4. **`Bytes<N>` narrowing casts disallowed.** `as Bytes<31>` from `Bytes<32>` is rejected at compile time.
5. **Renamed standard library functions.** `persistent_hash` ŌåÆ `persistentHash`, `hash_to_curve` ŌåÆ `hashToCurve` (camelCase migration between SDK versions).
6. **Wrong `CompactType` export name.** `CompactTypeCurvePoint` is not exported; the correct name is `CompactTypeNativePoint`.

#### Stealth wallet hypotheses

Two hypotheses were tested for how to enable a wallet to detect and spend a shielded note at the stealth address:

- ­¤¦¬ **HYPOTHESIS A** ŌĆö `ZswapSecretKeys.fromSeed(sk_bytes)` where `sk = (k_spend + h) mod JUBJUB_R`. Expected and confirmed **FALSE**: `fromSeed` applies a domain-separated KDF (`hash("midnight:csk" || seed_bytes)`) before producing the spending scalar, so `coinPublicKey ŌēĀ sk┬ĘG`.

#### Final stealth seed construction (two-ECDH)

After Hypothesis B was confirmed, the single-ECDH construction was upgraded to preserve the scan/spend key separation:

```
S_scan  = r ┬Ę K_scan    (= k_scan_scalar ┬Ę R on Alice's side)
S_spend = r ┬Ę K_spend   (= k_spend_scalar ┬Ę R on Alice's side)
stealthSeed = sha256(persistentHash(S_scan) || persistentHash(S_spend))
```

`stealthSeed` requires knowledge of **both** private scalars. A view-key holder of `(k_scan_scalar, K_spend)` can detect payments via the unshielded escrow path (compute `P = K_spend + h┬ĘG`, match against `pending_P`) but cannot derive `stealthSeed` and cannot spend shielded coins.

#### Privacy analysis

| Information | Unshielded escrow | Shielded path |
|---|---|---|
| Recipient name | Public | Public |
| Recipient public keys | Public | Public |
| Sender identity | Private | Private |
| Token amount | **Public** | Private |
| One-time stealth address | **Public** | Private |
| Payment event / timing | **Public** | **Public** (via `shielded_R`) |
| SenderŌĆōrecipient linkability | Direct | Indirect |

The registered name is the primary privacy trade-off: the keys themselves are opaque curve points, but the name is a durable long-term identifier regardless of path. Both paths leak payment timing through the published ephemeral key R. The shielded path hides amount and stealth address; the unshielded path exposes both.

### ­¤æ▒­¤ż¢ Throughput Model v1: Slot Timing Model and Constraint Solver Prospects

Developed working notes for a formal throughput model for Midnight and NEAR, exploring the structure of a slot timing model, critiquing its limitations, and assessing the viability of a constraint-solver approach to design-space exploration. Full notes in [artifacts/throughput-model-v1.md](../artifacts/throughput-model-v1.md).

- Defined a two-term timing model $\Delta_\text{slt} = \Delta_\text{diff} + \Delta_\text{ver}$ with verification duty cycle $\rho_\Delta = \Delta_\text{ver}/\Delta_\text{slt}$ connecting to $\gamma_\text{blk}$; the model fits Midnight/AURA cleanly due to AURA's deterministic slot assignment.
- Identified and characterised seven critiques of the two-term model with Midnight-specific analysis: (1) $\Delta_\text{diff}$ is a function of block size, not a free parameter ($\Delta_\text{diff} \approx 700$ ms at 5 MB); (2) $\Delta_\text{prod}$ is a missing third term (0.2% slot error at current TPS, growing to 8.6% at 500 TPS with 2 s slots); (3) streaming verification is impossible for Midnight's ZK proofs, making the sequential model a closer approximation than for UTXO chains; (4) $\Delta_\text{diff}$ is a probabilistic bound whose tail directly delays GRANDPA votes from slow validators; (5) Midnight's GRANDPA finality operates on a separate timescale from AURA block production, creating a two-timescale subtlety not captured by $\rho_\Delta$; (6) $\Delta_\text{prod}$ conflates NEAR's Turing-complete re-execution with Midnight's fixed-cost ZK verification.
- Extended to the three-term model $\beta_\text{txs} = \rho_\beta B / (\Delta_\text{prod} + \Delta_\text{diff} + \Delta_\text{ver})$ with a table mapping each slot term to its metric ($\gamma_\text{txs}$, $\Delta_\text{idle}/\beta_\text{blk}$, $\gamma_\text{blk}$ respectively); noted that $\beta_\text{txs}$ is non-linear in $B_\text{block}$ once each term's dependence is accounted for.
- Showed that substituting $\text{TPS} = \beta_\text{txs}/s_\text{tx}$ into the three-term identity yields a quadratic in $\beta_\text{txs}$ with a closed-form positive root; the two limiting cases reveal that $\beta_\text{txs} \propto B_\text{block}/d$ when network-dominated and $\propto \sqrt{B_\text{block}/a}$ when compute-dominated ŌĆö folding reduces bandwidth sensitivity as a side effect.
- Assessed TCP/QUIC complexity: $\Delta_\text{diff}$ reduces to $h(B/W + \ell) + \delta_\text{cwnd}$ where $\delta_\text{cwnd} = 0$ for QUIC or tuned TCP and $\approx 700$ ms for default Linux at 5 MB; at large block sizes QUIC offers <5% improvement in $\beta_\text{txs}$ but material benefits for finality latency (HoL-blocking isolation for GRANDPA messages) that belong in a separate finality model.
- Identified ten parameters sufficient for Midnight design-space exploration (three decisions: $B_\text{block}$, $T_\text{block}$, $k_\text{fold}$; five empirical: $W_{p67}$, $h$, $\ell$, $c_\text{proof}$, $s_\text{tx}$; one fixed: $c_\text{blk}$; one security: $\alpha$); noted that $W_{p67}$ (67th-percentile validator bandwidth) is the correct bandwidth parameter because GRANDPA requires >2/3 of stake-weighted validators to complete diffusion.
- ­¤¦¬ **HYPOTHESIS**: the model is sufficient accuracy for design-space exploration; dominant uncertainty is the unmeasured validator bandwidth distribution rather than transport protocol complexity; a constraint solver adds value for multi-objective optimisation and sensitivity sweeps but the primary feasibility questions are answerable from the closed-form quadratic.

### ­¤æ▒­¤ż¢ Throughput Constraints: NEAR Protocol vs. Midnight

Developed a comparative throughput metric framework for NEAR and Midnight and applied it to both chains. Full analysis in [assessments/throughput-constraint-comparison.md](../assessments/throughput-constraint-comparison.md).

- Defined a six-metric framework ŌĆö $\beta_\text{txs}$ (transaction-bytes/s), $\beta_\text{blk}$ (block-bytes/s), $\gamma_\text{txs}$ (per-transaction compute ms/s), $\gamma_\text{blk}$ (block-level fixed-overhead ms/s), $\Delta_\text{idle}$ (network idle time ms/s), $\pi_\text{shard}$ (shard parallelism) ŌĆö plus derived ratios $\rho_\beta = \beta_\text{txs}/\beta_\text{blk}$ (payload fraction) and $\rho_\gamma = \gamma_\text{txs}/\gamma_\text{blk}$ (per-transaction/block compute ratio, unbounded).

### ­¤ż¢­¤æ▒ Starstream and Nightstream: Architecture and Midnight Integration

Architecture and integration assessment of Starstream and Nightstream as the execution and proving layers of the MidnightOS vision, with analysis of their connection points to the current Midnight stack. - **Starstream.** A coroutine-based language (`.star` ŌåÆ WASM) whose `script fn` primitive maps to UTXO spending conditions; delimited continuations allow scripts to suspend at I/O boundaries and resume, making them composable with UTXO ledgers. The finalizer layer that converts ME obligations into a single blockchain-verifiable proof is not yet implemented.
- **Matched-Pair Relationship.** Starstream and Nightstream are intentionally coupled: Starstream produces execution traces shaped for Nightstream's CCS folding scheme, and together they form the core of the MidnightOS vision ŌĆö decentralized WASM applications delivered to the browser via a Docker-like protocol, with Starstream providing the execution context (including UI via `wasi-gfx`/WebGPU) and Nightstream proving integrity in the background.

- **Data Flow.** `.star` source compiles to WASM, runs locally producing public/private transcripts and an execution trace, feeds Nightstream IVC prover (chunked folding to a final proof), optionally routes through a Paima rollup sequencer for L2 batching, then settles to the Midnight L1 node for proof verification and ZSwap state update.
- **Security and Privacy.** The local-first privacy model is preserved unchanged ŌĆö private witnesses never leave the device and ZSwap's commitment/nullifier construction is unaffected ŌĆö and the integration adds post-quantum security and removes ZKIR's static circuit size limit. Two ­¤øæ blockers stand (`neo-midnight-bridge` finalizer engineering incomplete; no Starstream blockchain ABI or VM pallet), three ŌÜĀ’ĖÅ risks follow (bridge circuit performance unknown ŌĆö Goldilocks-in-BLS12-381 constraints are expensive and unbenchmarked; thinner audit coverage for lattice-based IVC; multi-proof-system tension with Paima/Kimchi), and a new architectural requirement emerges: a decentralized reputation-scored WASM delivery network (inspired by NEAR's BOS) to prevent malicious frontends from leaking private inputs.
- **Paima Synergy.** Nightstream IVC is the direct enabler of Paima Approach 3 (ZK-proven execution, Midnight as settlement) and simultaneously bypasses the Paima/Kimchi curve incompatibility ŌĆö if Nightstream is the prover, Kimchi is not needed and `neo-midnight-bridge` produces a Midnight-native BLS12-381 proof. ### ­¤ż¢­¤æ▒ NEAR Key Management and Key Structure

Deep-dive assessment of NEAR's key management capabilities, covering account model, named accounts, access key types, HD derivation, implicit accounts, multi-sig, Chain Signatures, and applicability to Midnight. Full analysis in [assessments/near-key-management.md](../assessments/near-key-management.md).

- **┬¦1 Account Model.** NEAR accounts maintain an on-chain map of public keys to access keys, supporting multiple simultaneous active keys as a first-class protocol feature rather than a contract abstraction. Account IDs may be human-readable named accounts, 64-hex Ed25519 implicit accounts, or `0x`-prefixed ETH-implicit (secp256k1) accounts.
- **┬¦2 Named Account Registration.** Account names are stored literally as UTF-8 bytes in trie keys with a 1-byte column discriminant prefix; the hierarchy is naming-only ŌĆö the parent has no ongoing authority over a sub-account after creation. There is no autonomous account garbage collection; zero-balance accounts persist indefinitely, and NEP-448 exempts small accounts (Ōēż 770 bytes) from the storage-staking floor entirely.
- **┬¦3 Access Key Permission Levels.** NEAR defines two permission levels ŌĆö FullAccess (12 action types) and FunctionCall (single-contract, optional method allowlist, optional spending cap) ŌĆö plus secp256k1 as a usable key type on named accounts at ~93 TGas ecrecover cost. DelegateAction meta-transactions bypass FunctionCall allowance checks since the relayer pays gas; all other FunctionCall validations (receiver, method, zero-deposit) still apply.
- **┬¦4 HD Key Derivation.** HD derivation is entirely a wallet-layer convention using BIP-39 + SLIP-0010 + BIP-44 (coin type 397); all NEAR HD paths are hardened because Ed25519's clamping invariant (`clamp(a+b) ŌēĀ clamp(a)+b`) destroys the linearity required for BIP-32 unhardened derivation. Cardano's BIP32-Ed25519 avoids this by multiplying the child tweak by 8, which preserves the low-three-bits-zero invariant and enables unhardened public-key derivation ŌĆö a more flexible but more complex scheme.
- **┬¦5 Implicit Accounts.** Ed25519 implicit accounts are latent 64-hex addresses activated on first token receipt; ETH-implicit accounts derive from keccak256 of the secp256k1 public key and automatically receive a Wallet Contract. Refund receipts do not trigger implicit account creation, creating a fund-burn risk if a refund is routed to a never-funded implicit address.
- **┬¦6 Key Registration.** `AddKeyAction` and `DeleteKeyAction` are explicit NEAR action types; key rotation is atomic because same-account actions are folded into one ActionReceipt on the account's shard (receipts, not transactions, are the unit of atomicity in NEAR). Contracts can manage keys on their own account only via promises (self-only constraint prevents unauthorized key injection), and the change lands in a subsequent block's receipt rather than the current one.
- **┬¦7 Multi-Sig and Key Rotation.** NEAR has no native threshold cryptography; m-of-n requires a smart contract (`near/core-contracts` multisig implements a confirmation flow with `add_request_and_confirm` / `confirm`). The NEAR web wallet's 2FA mode deploys a multisig contract and replaces the full-access key with a function-call key scoped to it, requiring second-device confirmation for high-value actions.
- **┬¦8 Chain Signatures.** Launched August 2024, Chain Signatures enables a NEAR account to control addresses on any blockchain via an 8-node threshold MPC network (`v1.signer`) using additive key derivation, making `(account_id, path)` deterministically yield stable foreign-chain addresses. Supported schemes are secp256k1 (domain_id=0) for EVM/Bitcoin and Ed25519 (domain_id=1) for Solana/Stellar/TON; Cardano is theoretically compatible via Ed25519 but unverified in practice; BLS12-381 is planned.
- **┬¦9 Additional Abstractions.** Meta-transactions (NEP-366) let users sign intents off-chain while relayers pay NEAR gas; FastAuth enables email-based key recovery without custodial key holding. All access keys are unconditionally public via RPC, leaking dApp engagement and usage patterns ŌĆö directly opposed to Midnight's privacy model. NEP-413 provides off-chain authentication via a domain-separated SHA-256 signature (`tag = 2^31+413`) restricted to FullAccess keys, with mandatory on-chain key verification as the critical server-side control step.

**┬¦10 Critical Assessment** ŌĆö evaluation of each capability across user value, corporate value, application areas, implementation locus, fundamental vs. cosmetic status, ecosystem fit, security posture, and complexity.

- **┬¦10.1 Named Account System.** NEAR's most visible UX innovation; the ASCII-only character set eliminates cross-script homoglyph attacks, but within-ASCII pairs (`0`/`o`, `1`/`l`) remain confusable. The naming hierarchy implies a governance relationship that does not exist at the protocol level ŌĆö `alice.near` has no authority over `evil.alice.near`.
- **┬¦10.2 FullAccess Keys.** Necessary primitive with no meaningful differentiation across chains; the NEAR-specific merit is making key management an explicit, auditable on-chain action rather than an implicit state write. Loss or compromise is catastrophic without application-layer safeguards.
- **┬¦10.3 FunctionCall / Session Keys.** NEAR's most significant and under-appreciated key management innovation; enables "login without custody" at the protocol layer with a genuine scoped blast radius. The zero-deposit restriction blocks the 1-yoctoNEAR security-deposit pattern common in NEP-141/NEP-171 standards, and allowance top-up requires delete-and-re-add.
- **┬¦10.4 DelegateAction / Meta-Transactions.** Decouples signer from gas payer, solving the onboarding bootstrapping problem; the relayer cannot alter the user's signed intent. The allowance bypass when a FunctionCall key signs a DelegateAction is intentional (relayer absorbs cost) but creates a subtle asymmetry.
- **┬¦10.5 Validator / Node Keys.** Pure infrastructure; the three-tier key separation (node key, validator key, account key) is sound ŌĆö a compromised hot validator node cannot drain the staking account. Inability to rotate the validator key during an active epoch without risking ejection is the primary operational security gap.
- **┬¦10.6 ETH-Implicit Accounts.** An explicit concession to Ethereum wallet compatibility; the automatic Wallet Contract and `rlp_execute` interface are a layered workaround with the lowest ecosystem coherence score. The hard prohibition on FullAccess keys and inability to delete the account are inelegant constraints.
- **┬¦10.7 Secp256k1 Access Keys.** Niche but real value for institutions with secp256k1 custody infrastructure wanting a named account; ~93 TGas ecrecover cost (one-third of per-receipt budget) makes secp256k1 keys unsuitable for high-frequency contract interactions.
- **┬¦10.8 HD Key Derivation.** Technically correct use of SLIP-0010 for Ed25519; all-hardened restriction precludes xpub-style public key derivation, limiting watch-only wallet patterns relevant to institutional custody. Single standardized path (`m/44'/397'/0'`) avoids ecosystem fragmentation.
- **┬¦10.9 Ed25519 Implicit Accounts.** Useful bootstrapping primitive for exchanges and faucets; the interaction between implicit accounts and refund receipts is a genuine fund-loss risk not obvious to developers. Two activation modes (Transfer for implicit, CreateAccount for named) and the refund-receipt exception complicate the mental model.
- **┬¦10.10 Promise-Based Key Management.** High value for recovery, DAO, and factory patterns; the self-only constraint is the crucial safety boundary preventing malicious key injection from a third-party contract. Deferred-receipt timing (key not visible until next block) can cause subtle race conditions in multi-step recovery flows.
- **┬¦10.11 Multi-Sig Contract.** Contract-layer multisig is functional but not philosophically distinguished; a bug in `near/core-contracts/multisig` affects all accounts using it. Chain Signatures provides a much stronger security model for high-value cross-chain custody.
- **┬¦10.12 Chain Signatures / MPC.** NEAR's most strategically significant capability ŌĆö no other L1 offers threshold MPC-based cross-chain signing as a production protocol primitive. The 8-node committee is small by BFT standards; the BN254/BLS12-381 curve incompatibility (NEAR's internal aggregation vs. Midnight's proof system) is the key blocker for any Midnight software adoption.
- **┬¦10.13 Key Enumeration.** Complete, unconditional public visibility of all access keys is a fundamental privacy deficit with no known mitigation path; FunctionCall entries reconstruct a detailed map of a user's dApp engagement history, usage frequency, and approximate login timestamps from the initial nonce value.
- **┬¦10.14 NEP-413 Signed Message.** Well-designed off-chain authentication standard; the domain separator (`2^31+413`) prevents signature reuse as a transaction, and the `recipient` field prevents cross-site replay. FullAccess-only requirement means daily-use authentication with a cold-stored root key is impractical.
- **┬¦10.15 FastAuth / Email Recovery.** Email-based recovery dramatically lowers onboarding friction but introduces a trusted relay network as a liveness and honesty dependency ŌĆö a principled tension between accessibility and trustlessness.

- **┬¦11.1 Named Accounts.** No Midnight equivalent; an off-chain ENS-style resolver is achievable at low cost but protocol-level named accounts are privacy-prohibitive ŌĆö they permanently link identity to address, undermining zero-knowledge selective disclosure before any proof is constructed.

- **┬¦11.3 FunctionCall / Session Keys.** No Midnight equivalent; the strongest Option 3 (Take Ideas) candidate ŌĆö the core security benefit (scoped blast radius) is directly applicable, but the entire design must be inverted: session key authorization in private Compact state, ZK proof of key validity inside the transition function, no on-chain revelation of `receiver_id` or `method_names`.
- **┬¦11.4 DelegateAction / Meta-Transactions.** No Midnight equivalent; near-term off-chain priority required for any consumer DApp deployment regardless of the NEARFall path, given Midnight's DUST fee bootstrapping problem. Privacy implications of relay payload visibility require a dedicated design review before committing to a standard.
- **┬¦11.5 HD Key Derivation.** Wallet-layer only, very low effort; Cardano's BIP32-Ed25519 is preferred over SLIP-0010 because it supports unhardened derivation and enables privacy-preserving diversified addresses (one fresh derived address per interaction). - **┬¦11.6 Implicit Accounts.** No Midnight equivalent; the NEAR design (address = public key) is structurally incompatible with Midnight's address-key unlinkability goal. A privacy-preserving voucher contract is the appropriate onboarding alternative and does not require any protocol change.
- **┬¦11.7 Promise-Based Key Management.** No Midnight equivalent; strong Option 3 value for social recovery, but requires a ZK redesign ŌĆö guardian relationships and recovery conditions held in private Compact state, with ZK proof of threshold satisfaction submitted without revealing guardian identities. Midnight's ZK model can improve on NEAR by keeping signing requests in private contract state, never recording them on-chain. ### ­¤ż¢ Claude's summary of NEAR storage

- NEAR archival storage totals ~118 TB (115 TB cold + 3 TB hot), split using the split storage architecture introduced in nearcore 1.35.0.
- The dominant cost is **state snapshots, not transaction history**. NEAR stores a full snapshot of the state trie for every shard at every epoch boundary (~12 snapshots/day across 6 shards).
- The state trie includes all account balances, contract code, and contract storage (key-value pairs). Popular contracts with many users can have hundreds of GB of state alone.
- Raw transaction/receipt data accounts for only ~5ŌĆō10 TB of the total ŌĆö roughly 1ŌĆō2 KB per transaction ├Ś 5.25 billion transactions.
- The remaining ~108ŌĆō113 TB is amortized state snapshot overhead, not transaction data.
- State snapshots are a **storage-for-latency tradeoff**: pre-computing and storing state at every epoch allows historical queries to be answered in milliseconds rather than requiring hours or days of replay.
- In principle, historical state could be reconstructed by replaying all transactions from genesis, but this is impractical for live RPC query serving.
- The hot/cold split mitigates cost: cold storage (~115 TB) can use cheap spinning disks or cloud persistent volumes since it is rarely queried; hot storage (~3 TB SSD) serves recent epochs where fast random access matters.
- Testnet archival storage is ~16 TB total (1.5 TB hot + 15 TB cold) ŌĆö much smaller due to lower contract activity and state accumulation.
- Total transactions on mainnet: ~5.25 billion, giving a ratio of ~22 KB of archival storage per transaction, of which only ~1ŌĆō2 KB is the transaction itself.
- As transaction volume grows, the per-transaction ratio will fall, since state snapshot cost grows with new accounts and contracts rather than proportionally with every transaction.
- A ZK-proof future could fundamentally change these economics by replacing full state storage with compact validity proofs.

#### ­¤ż¢ Notes by Gemini NotebookLM

- **Promises and Cross-Contract Calls:** NEAR provides a developer-friendly experience by allowing developers to use promises, which the runtime translates behind the scenes into specific messages passed between shards.
- **Action vs. Data Receipts:** The runtime primarily uses two types of receipts.
	- **Action receipts** carry instructions like creating an account, deploying a contract, or calling a function.
	- **Data receipts** do not contain actions; they solely carry the execution results (or a null/void result) of a contract, which are used to fulfill callbacks.
- **Data IDs for Callbacks:** When a contract requests a callback, an Action receipt is created but stored in the state rather than executing immediately. It waits for a specific **Data ID** to arrive via a Data receipt from the completed contract. This ID-based system allows callbacks to easily wait on multiple parallel executions (like waiting for two different token locks to succeed before executing a swap) without passing complex routing data back and forth.
- **Access Keys and Nonces:** NEAR accounts can have multiple access keys, which can either grant full permissions or restrict the user to specific function calls and gas allowances. To prevent transactions from colliding when multiple users operate the same account, nonces are recorded per access key rather than per account.
- **Account Deletion and Beneficiaries:** NEAR uses a "state staking" model where tokens are locked to pay for state storage. When an account is deleted, these tokens are unlocked, so the deletion action requires a "beneficiary" address to receive the funds. If a user provides an invalid beneficiary, the tokens are permanently burned.
- **Genesis State and Preserving Receipts:** The `apply_genesis_state` method runs once at the blockchain's start to set up initial accounts and contracts. It is also utilized during network forks to preserve unprocessed state elementsŌĆölike **delayed receipts** (queued due to congestion), **postponed receipts** (in-flight network messages), and partial **received data**ŌĆöensuring NEAR's strict guarantee that receipts are never lost.
- **Delayed Receipt Costs:** If executing local receipts hits the gas limit, the runtime converts the remaining ones into delayed receipts stored in the state. The system (and validators) absorbs the network cost of delaying these receipts to avoid penalizing contract developers with sudden, unpredictable fees.
- **The Callback Gas Limit Problem:** A unique challenge with asynchronous, sharded smart contracts is that if a callback function runs out of gas, its execution reverts, making the receipt effectively "lost" from the contract's perspective. To fix this, the team discussed implementing an event queue system where failed callbacks are recorded in the state, allowing anyone to send a new transaction to re-invoke the failed callback with more gas.

#### ­¤ż¢ Notes by Gemini NotebookLM

- **Sharded Architecture:** In NEAR, both execution and state are sharded, allowing nodes to avoid needing massive storage disks. Each shard has its own independent runtime that executes in parallel, functioning as a state machine. One step of this state machine represents the execution of one chunk (a piece of a block specific to that shard).
- **Transactions vs. Receipts:** The runtime primarily operates using two primitives.
    - **Transactions:** These originate from users outside the system via an RPC node and include a signature. As soon as a transaction enters the runtime, the necessary gas is deducted from the signer's account, and it is converted into a receipt.
    - **Receipts:** Receipts are essentially "trusted transactions" stripped of their signatures. They act as messages that shards exchange with one another (or themselves) to execute state changes.
- **Batched and Atomic Actions:** A transaction or receipt contains a list of actions (e.g., create an account, deploy a contract, and initialize it) so users don't have to submit multiple separate transactions. The execution of a receipt is completely atomic; if a single action in the batch fails, the entire receipt is rolled back.
- **Removing Colocation Incentives:** Cross-contract calls are asynchronous. To prevent developers from manipulating the system by attempting to host multiple contracts on the same shard for synchronous execution, NEAR treats calls to contracts on the _same_ shard exactly like cross-shard callsŌĆöthey are sent as a receipt and executed in the next block.
- **The `apply` Method & Processing Order:** The majority of the runtime's core logic and specification is defined in the `apply` method. The strict order of execution in this method is:
    1. Convert transactions into receipts.
    2. Process **local receipts** (a speed optimization specifically for when an account calls itself).
    3. Process **delayed receipts**.
    4. Process **incoming receipts** received from other shards.
- **Guaranteed Execution:** It is a fundamental guarantee of the NEAR blockchain that receipts are never dropped, which is highly critical for DeFi applications. If the runtime exceeds the gas limit while processing a chunk, the remaining receipts are placed into a "delayed receipts" queue stored in the state try to be processed later.
- **Balance Invariants:** At the end of execution, the runtime runs a massive invariant assertion called `check_balance` to verify that all token balances match up and that money did not magically appear or disappear during the chunk's execution.

### ­¤æ▒­¤ż¢ A few novelties of NEAR

- **Multiple actions in a transaction.**
    - All actions in a batch must share the same signer and receiver.
    - Batch actions are atomic with each other ŌĆö if one action fails, all are rolled back.
- **Receipts are executed in subsequent blocks.**
    - Cross-shard receipts may take additional blocks to propagate.
- **Function calls can create additional receipts.**
    - This is the mechanism behind cross-contract calls, which execute asynchronously in later blocks.
- **Transactions are not fully atomic.**
    - A function call in a successful transaction may create receipts that fail.
    - Failed receipts trigger a refund receipt back to the original signer, but state changes from already-executed receipts are not rolled back.
    - Failure must be handled explicitly in contract logic.
- **Accounts can be explicitly deleted.**
    - Deletion is triggered by a `DeleteAccount` action, which transfers the remaining balance to a beneficiary.
    - Deleted account names can be re-created by anyone, which has security implications (key reuse, history confusion).
- **Accounts can be garbage collected.**
	- A zero-balance account can be deleted by network processes.
	- Funds can be lost (to the validators) if they are sent to a garbage-collected account.
- **Named accounts with a hierarchical namespace.**
    - Account IDs are human-readable strings (e.g., `alice.near`).
    - A parent account (e.g., `alice.near`) can create and fully control sub-accounts (e.g., `sub.alice.near`), enabling DAO-managed namespaces and app-specific accounts.
- **Access keys with distinct permission levels.**
    - Accounts can hold multiple keypairs simultaneously.
    - _FullAccess_ keys can perform any action.
    - _FunctionCall_ keys are scoped to a specific contract and method set, with an optional NEAR allowance for gas, enabling delegated signing without exposing the root key.
- **Contracts are account state.**
    - Deploying a contract means storing Wasm bytecode in an account's state.
    - Any account can have a contract, and a contract can redeploy or delete itself.
- **Storage staking.**
    - Accounts must maintain a NEAR balance proportional to their on-chain storage usage.
    - This creates economic pressure against state bloat and ties storage costs directly to the account holding the data.
- **Chunk-based sharding (Nightshade).**
    - A single logical block is composed of _chunks_, one per shard.
    - Validators are assigned to shards and process only their chunk, while the block header commits to all chunks.
- **Gas refunds.**
    - Unused gas is automatically refunded to the signer as a receipt, rather than being forfeited as on EVM chains.
- **Deterministic Wasm execution via gas metering.**
    - Every Wasm instruction is metered, providing deterministic and portable execution bounds without relying on timeouts.
- **Meta-transactions (NEP-366).**
    - A relayer can submit a transaction on behalf of a user and pay the gas.
    - The inner transaction carries the user's signature, enabling gasless UX without trusting the relayer with the user's key.
