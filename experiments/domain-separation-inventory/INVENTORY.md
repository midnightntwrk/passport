# Domain-Separation Inventory

The canonical inventory of hash domain-separation use sites across Midnight's
Compact standard library and the `midnight-ledger` Rust source. Generated by
`./run-all.sh` (the mechanical enumeration), then manually verified — the
tag ↔ site association, the untagged classification, and the spec
cross-reference each require reading the surrounding code.

## 2 · Distinct domain tags

25 distinct tags in shipped code: **19 `midnight:`**, **6 `mdn:`**. Only **4**
are hoisted to a named `const`; the rest are inline literals. **None** is
declared in any architecture document or MIP (see §4).

`midnight:` — `zswap-pk[v1]`, `zswap-cc[v1]`, `zswap-cn[v1]`,
`zswap-ciphertext`, `derive_token`, `dsk`, `csk`, `esk`, `dust:proof`,
`schnorr_challenge`, `field_hash`, `entry-point`, `binding-input[v1]`,
`hash-intent:`, `hash-output-instruction-unshielded:`,
`sig-claim_rewards_transaction:`, `contract-update:`, `kernel:nonce_evolve`,
`kernel:nonce_evolve/2`.

`mdn:` — `lh`, `cc`, `cn`, `dust:pk`, `dust:cm`, `dust:nul`.

Form is unsystematic: 4 carry a `[v1]` version suffix, 4 carry a trailing
`:`, ≥3 are NUL-padded to a fixed 32-byte width (`derive_token`,
`schnorr_challenge`, `entry-point`), 2 are exact-fit `[u8; 12]` consts
(`esk`, `dsk`), and the remainder are plain. No length, casing, or
versioning rule is applied consistently.

## 3 · Untagged / non-domain-literal sites

| Site (`file:line`) | Primitive | Note |
|--------------------|-----------|------|
| `coin/coin.rs:894` UserAddress | `persistent_hash` | **UNTAGGED** — `persistent_hash(vk.binary_vec())`, no domain prefix |
| `merkle_tree.rs:143,145` internal node | `transient_hash` | untagged by leaf/internal convention (leaf is tagged `mdn:lh`) — defensible, but undocumented |
| `dust.rs:199,1083,1490,1756` nonce evolution | `transient_hash` | hashes structural `(initial_nonce, seq, …)` tuples, no domain literal |
| `dust.rs:243` initial nonce | `persistent_commit` | `opening = intent_hash` (a value, not a domain literal) |
| `zswap/construct.rs:73`, `zswap/verify.rs:136` binding | `transient_commit` | `opening = 0u8` — binding commitment, no domain literal |
| `ledger/construct.rs:597,891`, `onchain-runtime/lib.rs:46` comm. commitment | `transient_commit` | `opening = communication_commitment_rand` (random, not a domain) |

The opening-based commitments are not necessarily *wrong* — a value or random
opening can be the intended separation — but they are not domain-tagged, so a
caller cannot tell "untagged by design" from "tag forgotten" without reading
the construction. `UserAddress` is the clearest case of a missing prefix.

## 4 · Cross-codebase and spec divergences

These are the headline findings — the same conceptual objects carry different
domain separators across the implementation, the canonical specification, and
the specification's own diagram.

1. **Coin commitment / nullifier — Compact vs ledger.** The Compact stdlib
   tags the coin commitment/nullifier `mdn:cc` / `mdn:cn`
   (`standard-library.compact:229,241`); the ledger tags them
   `midnight:zswap-cc[v1]` / `midnight:zswap-cn[v1]` (`coin/coin.rs:628,639`).
   **Open question:** do these compute the same on-chain object (then this is
   a latent mismatch) or two distinct commitment families (then it is "merely"
   an inconsistent convention)? Either way it is unregistered and uncaught.

2. **Coin public key — implementation, spec, and diagram all disagree.**
   Ledger: `midnight:zswap-pk[v1]` (`coin/coin.rs:163`). Canonical WalletEngine
   Specification text and the diagram *source* `zswap-keys.puml`: `ni-pk[v1]` /
   `ni`. The *rendered* diagram `zswap-keys.svg` is stale against its own
   `.puml` and shows `mdn:pk`. Three separator values for one derivation —
   `midnight:zswap-pk[v1]` vs `ni-pk[v1]` vs `mdn:pk` — including a
   source-vs-rendered mismatch inside the spec itself.

3. **Key derivation — spec vs code.** The WalletEngine Specification derives
   `csk` / `esk` / `dsk` with domain separator `"ni"`; the ledger uses
   `midnight:csk` / `midnight:esk` / `midnight:dsk`
   (`zswap/keys.rs:60,69`, `dust.rs:204`).

4. **Two schemes inside the ledger.** `midnight:` governs zswap, intents,
   keys, and the kernel; `mdn:` governs dust and the Merkle leaf — and Dust
   itself mixes both (`midnight:dsk`, `midnight:dust:proof` alongside
   `mdn:dust:pk` / `:cm` / `:nul`).
